# i18n / Translation Support Design

**Date:** 2026-03-17
**Issue:** [#44 — Support multiple language translations](https://github.com/scuba10steve/steves-simple-storage/issues/44)
**Status:** Draft

---

## Overview

Add community-driven translation support to Steve's Simple Storage (and the s3-advanced companion mod). All user-facing strings already use `Component.translatable()` with keys in `en_us.json` — the infrastructure is correct. This spec covers the workflow, tooling, and artifacts needed to accept and validate contributed translations.

---

## Scope

### In scope

- `es_es.json` skeleton files for both `s3` and `s3_advanced` (all keys present, English stubs) as the first translation target
- Custom Gradle task (`validateLang`) that enforces key-set parity between any `*.json` lang file and its corresponding `en_us.json`
- `validateLang` hooked into the `check` lifecycle so CI catches mismatches on every PR
- GitHub PR template for translation submissions (`.github/pull_request_template.md`)
- `docs/translations.md` — contributor-facing guide explaining the full process
- Update to issue #44 body with a language checklist organized by family

### Out of scope

- Crowdin or any external localization platform (documented as a future path in `docs/translations.md`)
- Machine-translating any languages
- Modifying `en_us.json` (it remains the canonical source of truth)

## Artifacts

### Lang skeletons

Two files, one per module:

- `core/src/main/resources/assets/s3/lang/es_es.json`
- `neoforge/s3-advanced/src/main/resources/assets/s3_advanced/lang/es_es.json`

All keys copied from `en_us.json`. Values left in English with a leading `[TRANSLATE]` marker so it is immediately obvious which strings still need translation.

### Gradle validation task

The project uses Groovy DSL throughout (`build.gradle`, no `.kts` files). A task is registered in each relevant module's own `build.gradle` — not in the shared `s3.conventions.gradle` convention plugin, since not all modules have lang files (e.g., gametest modules do not).

The two modules that receive this task are `core` (namespace `s3`) and `neoforge/s3-advanced` (namespace `s3_advanced`). The `neoforge/s3` module has no assets directory — all user-facing strings for that module are packaged from `core` via the `coreClasses` configuration (`jar { from configurations.coreClasses }`) — so it is intentionally excluded from the task.

```groovy
tasks.register('validateLang') {
    group = 'verification'
    description = 'Validates all lang files have the same key set as en_us.json'

    // Replace 's3' with 's3_advanced' in neoforge/s3-advanced/build.gradle
    def langDir = file('src/main/resources/assets/s3/lang')
    inputs.dir(langDir)          // configuration-phase: enables up-to-date checking
    outputs.upToDateWhen { true } // no file outputs; skip if inputs unchanged

    doLast {
        def reference = new groovy.json.JsonSlurper().parse(new File(langDir, 'en_us.json'))
        def referenceKeys = reference.keySet()

        langDir.listFiles()
            ?.findAll { it.name.endsWith('.json') && it.name != 'en_us.json' }
            ?.each { langFile ->
                def keys = new groovy.json.JsonSlurper().parse(langFile).keySet()
                def missing = referenceKeys - keys
                def extra = keys - referenceKeys
                if (missing || extra) {
                    def msg = "${langFile.name} key mismatch:"
                    if (missing) msg += "\n  Missing: ${missing}"
                    if (extra) msg += "\n  Extra:   ${extra}"
                    throw new GradleException(msg)
                }
            }
    }
}

tasks.named('check') { dependsOn('validateLang') }
```

`inputs.dir` and `outputs.upToDateWhen` must be declared at the task configuration level (outside `doLast`) so Gradle evaluates them during the configuration phase and can skip the task when lang files are unchanged.

**Note for implementers:** Verify that `./gradlew build` triggers `check` in the `core` and `neoforge/s3-advanced` modules. These modules use `net.neoforged.moddev` rather than the plain `java` plugin; confirm the `build → check` lifecycle dependency is preserved before assuming CI picks up `validateLang` automatically.

### GitHub PR template

GitHub supports multiple PR templates via `.github/PULL_REQUEST_TEMPLATE/*.md`, but they require contributors to select them via a `?template=` URL query parameter — they are not surfaced automatically. For a small project where translation PRs are occasional, the simpler approach is a **single PR template** at `.github/pull_request_template.md` that applies to all PRs.

The template includes a translation section (conditionally applicable) with this checklist:

- Correct locale filename (e.g., `de_de.json`, not `de.json`)
- All keys present (CI will catch this, but self-check first)
- No keys added or removed relative to `en_us.json`
- Translation method disclosed (human, machine-assisted, or machine-only)
- Tested in-game with the target locale (encouraged, not required)

If a general PR template already exists, the translation checklist is added as an optional section within it. If no template exists, one is created with the translation section as the primary content.

### docs/translations.md

Contributor-facing documentation covering:

1. Finding the locale code for their language (links to Minecraft's locale reference)
2. Copying `en_us.json` to a new file named for their locale (e.g., `de_de.json`) — **not** copying from `es_es.json` or any other already-translated file, which would carry over translated values in the wrong language
3. Replacing each English value directly with the translation in their target language (no special prefix needed — just translate every value)
4. Running `./gradlew validateLang` locally to verify no keys are missing or extra
5. Opening a PR using the translation PR template

> **Note on `[TRANSLATE]` markers:** The committed `es_es.json` skeleton uses `[TRANSLATE] <English>` values to make untranslated strings visually obvious in-game. Contributors translating a new language copy from `en_us.json` and replace English values directly — they do not need to use the `[TRANSLATE]` marker.

> **Future:** Crowdin may be adopted if contribution volume grows. Any Crowdin-generated files must still pass `validateLang`.

## CI Integration

The existing GitHub Actions workflow runs `./gradlew build` (which includes `check`). Because `validateLang` is hooked into `check`, no separate CI job is needed — lang validation runs automatically on every PR.

If a lang file is missing keys or has extra keys, the build fails with a message listing the specific offending keys by filename.

## Future: Crowdin

If community contribution volume grows, Crowdin is the recommended next step. It provides a web UI for translators (no git knowledge required), a GitHub integration that auto-opens PRs when a language reaches a completion threshold, and is free for open source projects. The `validateLang` Gradle task would remain as a local verification tool even with Crowdin in place.
